# Niche Finder

**목적:** 파워로우 시장에서 수요의 정교한 사전 예측은 기대가치가 낮다. 이 스킬은 수요를 *예측*하지 않는다. **"내가 값싸게 도달할 유통 채널이 있는 좁은 틈"**을 무료 신호로 찾아 **테스트할 가설**로 내놓는다. 검증은 시장(출시 후 지표)이 한다. 유료 매출 추정 데이터는 쓰지 않는다.

## 세 가지 규율 (LLM 리뷰 마이닝 한계 방어 — 필수)

1. **정량 앵커 강제.** 모든 주장에 숫자(평점 수·경쟁 밀도·리뷰 속도). 숫자 없는 "기회"는 버린다.
2. **원문 인용 강제.** 페인포인트는 실제 리뷰 원문으로만. 지어내지 않는다(검증기가 대조).
3. **출력은 가설이지 검증된 수요가 아니다.** `verdict.type`은 항상 `"hypothesis"`, confidence는 low/med만. 리뷰는 *살아남은* 앱에서만 나온다(생존 편향) — "불만=기회" 단정 금지.

## 절차

1. **시드 확정.** 사용자에게 카테고리 또는 시드 키워드를 받는다(예: `["study timer","pomodoro"]`). 좁을수록 좋다 — 넓은 볼륨전 금지.
2. **무료 신호 수집.** 키워드마다 `node scripts/aso-probe.mjs "<keyword>" us 10` (skill 디렉토리 기준).
   - Search API(신뢰): `ratingCount`(설치 규모 프록시), 경쟁 밀도, 평점, 가격, 최신성.
   - 리뷰 RSS(best-effort): 리뷰 증가속도, 낮은 별점 원문. **빈 피드면 리뷰 신호 없이 진행**(데이터 없음 ≠ 불만 없음).
   - 검색 자동완성 수요는 브라우저 도구로 App Store 검색창을 직접 프로브(선택).
3. **유통 우위 판정.** 각 틈에 *내가* 값싸게 도달할 채널이 있는가 — 커뮤니티, 롱테일 SEO, 저경쟁 ASO 키워드. 유통 앵글 없으면 탈락.
4. **가설 작성.** `references/output-schema.md` 계약대로 JSON 생성. `quantAnchors.topApps`는 probe 출력에서 **그대로** 옮긴다(검증기가 Lookup 재대조). `reviewQuotes`는 실제 원문 ≥3. `dupRisk43`은 기존 앱·다른 가설과 바이너리·컨셉·메타데이터가 실질적으로 다른 근거와 함께 all-true(Apple 4.3 스팸 리젝 가드). 배열 순서 = 우선순위.
5. **자기검증 = checker (필수).** 스킬은 자기 산출물을 통과시키지 못한다: `node scripts/aso-verify.mjs <output.json>` → exit 0이어야 진행, 1이면 위반 고치고 재생성.
6. **하류 연결.** 사용자가 가설 1개를 고르면: 프로젝트에 스캐폴드 파이프(스코어카드·PRD 템플릿·new-app 류)가 있으면 가설의 차별점·페인포인트를 그 파이프에 기입해 넘긴다. 없으면 **검증 통과한 가설 JSON이 최종 산출물**이다.

## 출시 후 실측 (checker B — 최종 채점)

스킬은 가설만 낸다. 최종 판정은 출시 지표(노출→전환·키워드·리텐션 등 스킬이 못 만드는 데이터)가 내린다. 가설 → 착수 → 출시 지표 → 적중/오답을 프로젝트의 가설 원장에 기록해 다음 픽의 피드백으로 쓴다.

## 유료 API (기본: 안 씀)

매출 추정치(AppFigures 등)는 **연납 금지**. 특정 결정이 요구할 때만 월납 1개월 → 필요한 데이터만 뽑고 해지. 조건은 해당 가설의 `paidApiTrigger`에 적는다.
